PART 4 — Idempotency protection

Implement idempotency keys for every mutating job.

Examples:

* send_email:{orgId}:{workflowRunId}:{stepId}:{recipient}
* send_sms:{orgId}:{workflowRunId}:{stepId}:{phone}
* book_session:{orgId}:{athleteId}:{slotId}
* workflow_step:{orgId}:{workflowRunId}:{stepId}
* memory_lifecycle:{orgId}:{date}

Rules:

* If a completed job exists with the same idempotencyKey, do not execute again.
* Return the previous result.
* Log duplicate prevention to unified_agent_action_log.
* Never double-send communications.
* Never double-book sessions.
* Never double-charge billing actions.

⸻

PART 5 — Retry + failure classification

Create structured failure handling.

Failure types:

1. Transient
    Examples:

* network timeout
* provider temporarily unavailable
* database timeout

Behavior:

* retry with exponential backoff
* max attempts configurable by job type

2. Rate limited
    Examples:

* SendGrid rate limit
* Twilio rate limit
* internal org quota hit

Behavior:

* retry at provider reset time or calculated backoff

3. Governance
    Examples:

* emergency pause enabled
* agent permission revoked
* confidence below threshold

Behavior:

* pause job
* send to approval/attention queue
* do not retry automatically until governance changes

4. Blocked
    Examples:

* missing email
* missing phone
* disconnected integration
* incomplete account setup

Behavior:

* create attention item
* mark paused/blocked
* allow manual resume

5. Fatal
    Examples:

* invalid workflow definition
* deleted entity
* impossible state

Behavior:

* move to dead letter
* create critical attention item

PART 10 — Workflow execution migration

Upgrade existing workflow execution so:

Current:
workflow runs execute directly/synchronously.

New:
workflow runs enqueue jobs.

Behavior:

* starting a workflow creates initial workflow_step job
* each completed step enqueues the next step
* approval gates pause job execution until approval
* rejected jobs stop or reroute
* regenerated jobs enqueue new draft generation
* completed workflows record workflow_outcomes
* all transitions write to unified_agent_action_log

Keep old execution behavior as fallback behind a feature flag:
USE_WORKFLOW_JOB_QUEUE=true

PART 14 — Reliability safeguards

Critical safeguards:

* No duplicate sends
* No duplicate bookings
* No duplicate workflow steps
* No double billing
* All mutating operations require idempotency keys
* All jobs are org-scoped
* All locks expire
* All retries are bounded
* All failures are classified
* Emergency pause stops future job execution safely
* Already-running jobs finish safely unless critical stop is required
* Manual retry must re-check governance before execution
