Audit and fix the entire TrainEfficiency password reset flow.

Problem:
A user requested a password reset, but the email link opened a Replit preview domain instead of the production TrainEfficiency domain. The page showed “Run this app to see the results here,” meaning the reset link is using the wrong base URL and the reset process may be broken.

Goal:
Make password reset work reliably in production using the TrainEfficiency public domain, not any Replit/JaneWay preview URL.

Audit these areas:

1. Password reset request endpoint
Find the backend route that handles “forgot password” / “reset password request.”
Check how the reset URL is generated.
It should NOT use:
- req.headers.host
- origin from the incoming request
- Replit preview URLs
- localhost
- dynamic preview domains

It should use a stable environment variable, for example:

PUBLIC_APP_URL=https://trainefficiency.com

or whatever the real production TrainEfficiency domain is.

The reset link should be generated like:

`${PUBLIC_APP_URL}/reset-password?token=${token}`

or whatever the frontend route expects.

2. Environment variables
Add or verify:

PUBLIC_APP_URL=https://trainefficiency.com

If the app uses a different production domain, use that exact domain.

Also ensure backend email templates never fall back to Replit domains in production.

3. Frontend reset route
Audit the reset-password page route.

Confirm:
- The route exists.
- It works on direct navigation.
- It can read the token from the URL.
- It validates the token or submits the token with the new password.
- It shows proper loading/error/success states.
- After success, it sends the user back to login.

4. Token handling
Audit the reset token logic.

Confirm:
- Token is securely generated.
- Token is hashed before storing if that is the current auth pattern.
- Token has an expiration time.
- Token is single-use.
- Expired/invalid tokens return clear errors.
- Successful reset clears the token fields.

5. Email template
Audit the password reset email.

Confirm:
- Link uses the production TrainEfficiency domain.
- Button href and plaintext fallback URL match.
- Email copy clearly says the link expires.
- No Replit URL appears anywhere in the email.

6. Mobile behavior
Because the reported bug happened from Gmail on iPhone, make sure the reset link opens cleanly on mobile Safari/Gmail in-app browser.

7. Safety checks
Add logging that records the generated reset URL domain only, not the token.
Example:
console.log("[password-reset] Generated reset link domain:", new URL(resetUrl).origin)

Do not log full reset tokens.

8. Tests/manual verification
Add or update tests if the project has tests.

At minimum verify manually:
- Request reset from login page.
- Receive email.
- Email link uses TrainEfficiency domain.
- Link opens reset-password page directly.
- New password can be submitted.
- User can log in with new password.
- Same reset link cannot be reused.
- Expired/invalid token shows a clean error.

Important:
Do not break existing login, registration, JWT/session handling, or organization/client portal auth.

Return:
- Files changed
- Root cause
- Exact environment variable required
- Confirmation that no Replit domain is used in generated password reset emails
- Manual QA checklist
