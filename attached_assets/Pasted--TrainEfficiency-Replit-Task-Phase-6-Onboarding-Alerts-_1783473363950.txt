# Core Alert Conditions

Create alert logic for incomplete onboarding records.

Detect these conditions:

## 1. Welcome Draft Stuck

Condition:

* `welcomeDraftQueued = true`
* `welcomeDraftApproved = false`
* checklist created more than 24 hours ago

Alert:

> Welcome draft has not been approved for [Athlete Name].

Severity:

* Medium after 24h
* High after 72h

---

## 2. First Session Not Scheduled

Condition:

* `firstSessionScheduled = false`
* checklist created more than 24 hours ago

Alert:

> First session has not been scheduled for [Athlete Name].

Severity:

* High after 24h
* Critical after 72h

---

## 3. Program Not Assigned

Condition:

* `programAssigned = false`
* checklist created more than 24 hours ago

Alert:

> No training program has been assigned to [Athlete Name].

Severity:

* Medium after 24h
* High after 72h

---

## 4. Guardian Missing

Condition:

* lead submission has `parentEmail`
* `guardianLinked = false`
* checklist created more than 24 hours ago

Alert:

> Parent/guardian account has not been linked for [Athlete Name].

Severity:

* Medium

---

## 5. Onboarding Stuck

Condition:

* `status != "complete"`
* checklist created more than 3 days ago

Alert:

> Onboarding has been open for [N] days for [Athlete Name].

Severity:

* High after 3 days
* Critical after 7 days

---

## 6. Optional Billing / Waiver Blockers

Only activate these if the app has existing billing or waiver support enabled.

Do not make fake alerts for systems that do not exist.

Possible alerts:

* Payment setup incomplete

# Backend Scope

## 1. Central Alert Helper

Create a reusable helper/service for onboarding alerts.

Suggested file:

```txt
server/services/athlete-onboarding-alerts.ts
```

Responsibilities:

* Accept onboarding checklist + lead + athlete context.
* Return normalized alert objects.
* Compute severity.
* Compute age in hours/days.
* Avoid duplicate alert types per athlete.
* Produce stable alert keys.

Example return shape:

```ts
{
  key: "onboarding:first_session_missing:<checklistId>",
  type: "first_session_missing",
  severity: "critical" | "high" | "medium" | "low",
  title: "First session not scheduled",
  message: "First session has not been scheduled for Marcus Smith.",
  athleteUserId,
  checklistId,
  leadSubmissionId,
  ageHours,
  actionLabel: "Schedule Session",
  actionUrl: "/admin/scheduling-command-center",
  createdAt,
}
```

## 6. Alert Resolution

Alerts should disappear automatically when the checklist is updated.

Examples:

* If `firstSessionScheduled = true`, first-session alert disappears.
* If `programAssigned = true`, program alert disappears.
* If `welcomeDraftApproved = true`, draft alert disappears.
* If `guardianLinked = true`, guardian alert disappears.
* If status becomes complete, stuck alert disappears.

No manual resolve UI required in this phase unless an existing pattern exists.

# Data Rules

## Avoid Alert Spam

Only one alert per type per checklist.

Do not create multiple alerts for the same missing program every page load.

Prefer computed active alerts unless an existing Attention Inbox requires persistence.

## Severity Rules

Use deterministic severity rules.

Do not use AI to classify severity in this phase.

## Alert Age

Use checklist `createdAt` or `updatedAt` carefully:

* Missing initial onboarding steps should age from checklist `createdAt`.
* Stuck onboarding should age from checklist `createdAt`.
* If a step was manually updated, do not reset the entire checklist age unless the schema supports per-step timestamps.
