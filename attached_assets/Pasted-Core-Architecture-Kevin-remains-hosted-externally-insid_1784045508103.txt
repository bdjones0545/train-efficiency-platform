Core Architecture
Kevin remains hosted externally inside his virtual computer.
TrainEfficiency remains the authoritative execution environment.
Kevin must never receive direct unrestricted database access. He must operate through authenticated, organization-scoped TrainEfficiency APIs and registered capabilities.
Implement this flow:
User or System Event
        ↓
Kevin receives context
        ↓
Kevin creates an operational intent
        ↓
TrainEfficiency validates identity, organization and capability
        ↓
Policy engine determines execution mode
        ↓
Action executes, becomes a draft, or requires approval
        ↓
Result is verified
        ↓
Outcome is recorded
        ↓
Kevin receives the result and updates executive memory

Phase 3 — Policy and Permission Engine
Every Kevin action must pass through a centralized policy engine.
Validate:
Kevin service identity
organization membership
organization status
user initiating the request
user role
capability availability
capability mode
requested resource ownership
organization isolation
consent requirements
risk classification
approval requirements
rate limits
idempotency
current circuit-breaker state
Map capability modes to behavior:
disabled          → reject
observe           → return available data only
recommend         → generate a recommendation without creating an action
draft             → create a reversible draft
require_approval  → create an approval request and stop
auto              → execute within policy limits
Do not implement one global “Kevin can do everything” permission.
Permissions must be set per organization and per capability.
Add safe defaults. New capabilities should default to disabled or require approval, depending on risk.

Phase 5 — Intent and Execution Model
Introduce a durable Kevin intent model.
An intent represents the executive goal, while one or more tasks represent the actual actions required.
Example:
Intent:
“Follow up with five high-value leads that have been inactive for seven days.”

Tasks:
1. Ask Revenue Agent to rank the leads.
2. Ask AgentMail to create personalized drafts.
3. Create an approval bundle.
4. Send approved emails.
5. Record replies and revenue outcomes.
Suggested intent states:
received
validating

awaiting_approval
queued
executing
verifying
completed
partially_completed
failed
cancelled
dead_lettered
Persist:
intent
execution plan
individual tasks
dependencies
approvals
executor
attempts
outputs
verification results
final outcome
failure reason
timestamps
audit events
Use durable database-backed state, not an in-memory map.
Phase 6 — Agent Task Bus
Implement or extend a shared task bus through which Kevin delegates work to TrainEfficiency agents.
Kevin should not directly edit another agent’s memory or internal state.
Use this pattern:
Kevin
  ↓
Creates structured agent task
  ↓
Task bus routes task to specialist agent
  ↓
Agent accepts or rejects task
  ↓
Agent performs work
  ↓
Agent returns structured output
  ↓
Kevin reviews the output
  ↓
Kevin approves next step, escalates or closes intent
Each task must include:
assigned agent
organization ID
capability requested
objective
inputs
constraints
priority
due time
approval requirements
parent intent
expected output schema
status
attempts
final output
Add loop prevention so agents cannot indefinitely delegate tasks back and forth.
Add maximum delegation depth and correlation-chain tracking.
Phase 7 — CEO Agent Integration
Kevin should not blindly replace the existing CEO Agent.
Define a clear relationship:
Kevin is the persistent executive orchestrator.
The CEO Agent remains the TrainEfficiency-native executive interface and analysis layer.
Kevin may ask the CEO Agent for analysis, briefings, decisions and platform context.
The CEO Agent may escalate events or recommendations to Kevin.
Both should operate through structured tasks and outcomes, not hidden prompt injection or direct memory edits.
Implement capabilities such as:
ceo.request_analysis
ceo.request_briefing
ceo.request_decision
ceo.submit_recommendation
ceo.escalate_risk
Prevent circular calls between Kevin and the CEO Agent.
Phase 8 — AgentMail Integration
Connect Kevin to the existing AgentMail system.
Kevin must be able to:
request a new email draft
request a reply draft

request revisions
submit a draft for approval

Do not create a second email-writing system.
Use the existing AgentMail learning, prior-contact context, approval and outcome infrastructure.
High-risk email actions should default to:

Sending automatically should only be available when explicitly enabled for the organization and capability.

Phase 11 — Verification and Closed-Loop Learning
A successful API response is not enough to mark an action complete.
Every capability must define how its outcome is verified.
Examples:
Email draft: verify the draft exists and contains the expected recipients.
Email send: verify provider acceptance and delivery status when available.
Agent task: verify the agent returned output matching the schema.
Schedule creation: verify the event exists and has no conflict.
Lead update: verify the correct organization-owned lead changed.
Navigation action: verify the requesting user has route access.
After verification, record:
requested action
executed action
actual result
expected result
outcome difference
user approval or rejection
downstream business result
agent involved
Kevin confidence
whether the same action should be recommended in the future
Feed outcomes into the existing Kevin outcome and memory systems without storing sensitive raw data unnecessarily.
