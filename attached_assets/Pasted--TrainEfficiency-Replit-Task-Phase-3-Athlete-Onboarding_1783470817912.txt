---

# Goal

When a lead is converted into an athlete, the system should not create an empty athlete shell.

It should enrich the athlete record with useful onboarding context from the intake submission so coaches, PAIL, and future AI systems immediately understand:

* Who the athlete is
* What sport they play
* What school/grade they are in
* What their goals are
* What their training background is
* Whether a parent/guardian is involved
* What onboarding steps still need completion

---

# Scope

## 1. Seed Athlete Intelligence / PAIL Context

After conversion, seed athlete intelligence with intake context.

Use existing PAIL / athlete intelligence / context tables if they exist.

Look for existing tables or services such as:

* `athlete_context_objects`
* `athlete_memory`
* `athlete_intelligence_profiles`
* `athlete_learning_engine`
* `athlete_notes`
* `coach_notes`
* `performance_profiles`

Do not create duplicate systems if one already exists.

Seed structured context from:

* Sport
* Position
* School
* Grade
* Age
* Goals
* Experience level
* Current training status
* Commitment level
* Parent/guardian name
* Lead source
* Coach/admin notes

Expected behavior:

* New athlete receives useful context immediately.
* PAIL synthesis should not start from a completely blank state.
* Intake context should be clearly marked as coming from lead capture.
* Avoid overwriting future coach-entered data.

## 2. Guardian / Parent Linking

Audit existing guardian/parent schema.

Look for:

* `athlete_guardian_links`
* `guardians`
* `parents`
* `family_members`
* `emergency_contacts`
* `user_relationships`

If supported:

* Create or link guardian record from parent/guardian fields.
* Link guardian to athlete.
* Preserve parent name/email/phone if available.
* Avoid duplicate guardian records.
* Do not invent a full parent portal if none exists.

If not supported:

* Store parent/guardian info as athlete context.
* Return `guardianLinked: false` with a clear reason.
