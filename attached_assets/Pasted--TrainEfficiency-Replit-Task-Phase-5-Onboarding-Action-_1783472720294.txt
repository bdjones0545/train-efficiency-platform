# Phase 5 Goal

Turn onboarding from a hidden checklist into an operational workflow.

Coaches/admins should be able to:

* View all converted athletes with incomplete onboarding
* See what each athlete still needs
* Take one-click actions on missing steps
* Filter by status
* Open the source lead, athlete intelligence, scheduling, and AgentMail draft
* Resolve stuck onboarding before athletes fall through the cracks

---

# Core Product Concept

Create an admin-facing onboarding command center.

Suggested route:

```ts
/admin/athlete-onboarding
```

Suggested sidebar label:

```txt
Athlete Onboarding
```

This should be operational, not cosmetic.

It should answer:

> “Which new athletes are not fully onboarded, and what do I need to do next?”

# Backend Scope

## 1. List Onboarding Records

Add a route such as:

```ts
GET /api/admin/athlete-onboarding
```

Return onboarding checklist records joined with:

* athlete user
* user profile
* linked lead submission
* guardian info if available
* welcome draft ID/status if available
* latest intelligence profile/stage if available
* recommended next actions if available

Support filters:

```ts
status=pending|in_progress|complete
needs=guardian|program|session|payment|waiver|welcome|account|pail
search=name|email|sport|school
```

Return useful fields:

```ts
{
  id,
  athleteUserId,
  leadSubmissionId,
  athleteName,
  email,
  phone,
  sport,
  school,
  grade,
  parentName,
  parentEmail,
  guardianLinked,
  accountInviteSent,
  welcomeDraftQueued,
  welcomeDraftApproved,
  pailContextSeeded,
  firstSessionScheduled,
  programAssigned,
  paymentSetup,
  waiverCompleted,
  firstSessionCompleted,
  nextBestAction,
  status,
  createdAt,
  updatedAt,
  recommendedNextActions,
  links: {
    lead,
    athleteIntelligence,
    aiApprovals,
    scheduling,
    athleteProfile
  }
}
```

---

## 2. Update Checklist Items

Add a route such as:

```ts
PATCH /api/admin/athlete-onboarding/:id
```

Allow admins to update checklist booleans manually when appropriate:

* paymentSetup
* waiverCompleted
* firstSessionCompleted
* programAssigned
* firstSessionScheduled
* welcomeDraftApproved

Rules:

* Recalculate `status` after every update:

  * `complete` if all required items are complete
  * `in_progress` if at least one step is complete
  * `pending` if nothing meaningful is complete
* Recalculate `nextBestAction`
* Do not allow unsafe mutation of identity fields from this route.

## 4. Recommended Next Action Logic

Create a small centralized helper that determines `nextBestAction`.

Priority order suggestion:

1. Account invite not sent
2. Welcome draft not queued/reviewed
3. PAIL context not seeded
4. Guardian not linked when parent email exists
5. First session not scheduled
6. Program not assigned
7. Payment setup incomplete
8. Waiver incomplete
9. First session not completed
10. Onboarding complete

This helper should be reusable by:

* conversion route
* onboarding list route
* checklist update route

# Data / Logic Rules

## Required vs Optional Items

Not every item should block completion for every org.

For Phase 5, use conservative defaults:

Required for complete:

* accountInviteSent
* welcomeDraftQueued
* pailContextSeeded
* firstSessionScheduled
* programAssigned

Optional / manual:

* guardianLinked only required if `parentEmail` exists
* paymentSetup only required if billing is enabled or if no safe billing detection exists, mark as optional/manual
* waiverCompleted only required if waiver system exists
* firstSessionCompleted should remain incomplete until after first session

If the existing app does not support billing/waiver yet, do not make onboarding impossible to complete because of those placeholders.
