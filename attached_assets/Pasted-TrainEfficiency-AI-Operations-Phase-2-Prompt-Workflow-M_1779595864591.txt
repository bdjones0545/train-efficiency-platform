TrainEfficiency AI Operations — Phase 2 Prompt

Workflow Memory + Context Persistence Layer

Build Phase 2 of the AI Operations architecture.

Goal:
Transform workflows and agents from stateless automations into memory-aware operational systems that learn from previous interactions, outcomes, and organizational behavior over time.

This phase should establish the foundation for:

* adaptive workflows
* persistent organizational intelligence
* contextual AI reasoning
* historical outcome awareness
* long-term operational memory
* future Business Brain learning loops

IMPORTANT:
Do NOT yet build autonomous learning or self-modifying agents.
This phase is strictly about:

* memory persistence
* context retrieval
* outcome tracking
* historical awareness
* workflow continuity
* explainability

The system must remain deterministic, auditable, and org-scoped.

PART 3 — Workflow Context Engine

Create:
server/workflow-context-engine.ts

Responsibilities:

* retrieve relevant memory/context before workflow execution
* compress historical interactions into usable summaries
* rank memory relevance
* attach contextual memory to workflow runs
* persist new memory after workflow completion
* maintain explainability

Functions:

* getWorkflowContext()
* buildContextSummary()
* persistWorkflowMemory()
* calculateMemoryRelevance()
* attachOutcomeToMemory()

The engine should:

* prioritize recent + high-impact context
* avoid infinite memory growth
* compress low-value historical events
* preserve important operator overrides permanently

⸻

PART 4 — Context-aware workflow execution

Upgrade workflow execution engine so every workflow run can receive:

* historical workflow context
* prior communication history
* previous outcomes
* operator feedback history
* prior approval edits
* known entity preferences
* AI reasoning summaries

Example:
If a lead ignored 3 previous email workflows but responded to SMS:
future workflows should receive this context.

Example:
If an admin repeatedly edits AI-generated outreach tone:
future draft generations should include this preference context.

* override AI decisions

operator_override memories.

Purpose:
Allow future workflows to align better with organizational preferences without becoming fully autonomous.

PART 7 — Business Brain memory integration

Upgrade Business Brain so recommendations can reference:

* historical recommendation outcomes
* prior accepted/rejected recommendations
* recurring operational patterns
* long-term organizational trends

Examples:

* “The last 3 retention campaigns using SMS outperformed email by 24%.”
* “This scheduling recommendation is similar to a prior successful intervention from March.”

Do NOT yet allow the Business Brain to self-modify strategy.
Only allow historical contextual awareness.

⸻

PART 8 — Memory lifecycle management

Implement memory lifecycle rules.

Requirements:

* low-value memories compress automatically
* duplicate memories merge
* stale low-importance memories archive
* high-importance memories persist indefinitely
* operator overrides never auto-delete
* auditability must remain intact

Add:
memoryImportanceScore calculation logic.

Factors:

* business impact
* operator interaction frequency
* workflow success correlation
* recurrence frequency
* escalation involvement

* No self-modifying prompts
* No hidden autonomous learning

PART 12 — Architectural goal

The goal of Phase 2 is NOT AGI or fully autonomous behavior.

The goal is:

* persistent operational intelligence
* adaptive workflow continuity
* historical awareness
* organizational preference memory
* explainable contextual automation

This phase should make TrainEfficiency feel like:
“the system remembers how this organization operates.”

The architecture should remain:

* auditable
* observable
* controllable
* explainable
* org-isolated
* operator-first
