Build a complete “Forgot Password / Reset Password” system for TrainEfficiency that works for ALL account types, including standard users, coaches, and admins if admins exist in the current auth model.

GOAL
Users must be able to securely reset their password from the sign-in screen without contacting support.

CORE REQUIREMENTS

1. ADD “FORGOT PASSWORD?” TO LOGIN
- On the login screen, add a clearly visible “Forgot Password?” link beneath the password field.
- This should open a dedicated forgot-password flow, not a placeholder.
- Maintain current TrainEfficiency branding and mobile responsiveness.

2. FORGOT PASSWORD PAGE / MODAL
Create a forgot-password screen with:
- headline: “Reset your password”
- supporting text: “Enter the email associated with your account and we’ll send you a password reset link.”
- email input
- submit button
- back to login link

Behavior:
- user enters email
- system always returns a neutral success message like:
  “If an account exists for that email, a password reset link has been sent.”
- never reveal whether the email exists in the system

3. PASSWORD RESET TOKEN SYSTEM
Implement a secure reset-token flow:
- generate a cryptographically secure random token
- DO NOT store raw token in database
- hash the token before storing
- save:
  - email / userId reference
  - hashed token
  - expiration timestamp
  - usedAt timestamp nullable
- token should expire after 1 hour
- each new reset request invalidates older unused reset tokens for that account
- token must be single use only

4. EMAIL DELIVERY
Send a password reset email containing a secure reset link like:
- https://your-domain/reset-password?token=XYZ

Email should include:
- clear subject line: “Reset your TrainEfficiency password”
- short explanation
- button/link to reset password
- expiration notice
- message saying if they didn’t request this, they can ignore the email

If email service already exists in the app, reuse it.
If not, create an email utility abstraction so providers can be swapped later.

5. RESET PASSWORD PAGE
Create a reset-password page that:
- reads token from query params
- validates token securely on backend
- if valid, shows:
  - new password field
  - confirm password field
  - show/hide password toggle
  - submit button
- if invalid or expired, show a clean error state with:
  - “This reset link is invalid or has expired.”
  - CTA to request a new link

6. PASSWORD RULES
Enforce strong password validation:
- minimum 8 characters
- at least 1 uppercase letter
- at least 1 lowercase letter
- at least 1 number
- optional special character encouragement if current auth allows it

Show inline validation errors in a polished way.

7. SUCCESS STATE
After successful reset:
- mark token as used
- update password using the same password hashing strategy as current auth system
- redirect to login page
- show success message:
  “Your password has been reset successfully. Please sign in.”

8. SECURITY REQUIREMENTS
This is critical:
- no user enumeration
- rate limit forgot-password requests by IP and by email
- hash stored reset tokens
- expire tokens automatically
- invalidate all prior reset tokens when a new one is created
- clear/reset active sessions if the current auth model supports session invalidation on password change
- add server-side validation for all steps
- never trust frontend-only validation

9. DATABASE / BACKEND WORK
Audit the existing auth schema and implement whatever is needed cleanly.
If no reset-token table exists, create one.

Suggested schema:
password_reset_tokens
- id
- user_id
- email
- token_hash
- expires_at
- used_at
- created_at

Make sure this works for all account roles without duplicating logic.
Use one unified password reset system across users, coaches, and admins if possible.

10. FRONTEND UX POLISH
Make the flow feel premium and frictionless:
- mobile friendly
- clear states for loading, success, and error
- disabled submit buttons while processing
- prevent duplicate submissions
- clean copy and spacing consistent with TrainEfficiency UI
- do not allow broken layouts on small screens

11. AUTH INTEGRATION
Audit current authentication code and integrate reset logic into the existing login system.
Do not create parallel auth systems.
Use the same password hashing/auth user source already used by TrainEfficiency.

12. EDGE CASES
Handle all of these:
- email entered with uppercase/lowercase differences
- multiple reset requests in a row
- expired token
- already used token
- invalid token
- user opens old email after requesting a newer reset
- password confirmation mismatch
- network/server failure with graceful UI messaging

13. TESTING
After building, verify:
- existing users can request reset
- existing coaches can request reset
- nonexistent emails still show neutral message
- reset email sends correctly
- token expires correctly
- used token cannot be reused
- password actually changes in DB
- old password no longer works
- new password works immediately
- login flow still works normally
- no role-specific auth regressions

14. DELIVERABLE
Implement the full feature, wire it into the app, and then provide:
- summary of files added/updated
- any migration created
- any environment variables required for email sending
- any manual setup steps still needed

IMPORTANT
- Build this as production-ready, not mock/demo only
- Reuse existing auth architecture
- Keep code organized and avoid spaghetti logic
- Prefer shared services/utilities for token creation, email sending, validation, and password updating
- Do not break current login/signup flows
