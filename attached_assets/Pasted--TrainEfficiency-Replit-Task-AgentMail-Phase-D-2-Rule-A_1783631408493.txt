# Goal

Track which learning rules and standing instructions are injected into each generated draft.

This unlocks real per-rule performance analytics.

The system should be able to answer:

* Which rules were used on this draft?
* Which rules correlate with approval?
* Which rules correlate with editing?
* Which rules correlate with rejection?
* Which standing instructions are actually useful?
* Which learned rules may be hurting performance?

# Scope

## 1. Return Rule IDs From Learning Context

Update:

```txt
server/services/message-learning-service.ts
```

Current:

```ts
getMessageLearningContext(orgId, communicationDomain)
```

It returns prompt text.

Update it so callers can also get structured metadata about the rules used.

Recommended:

```ts
getMessageLearningContextWithRules(orgId, communicationDomain)
```

Return:

```ts
{
  contextText: string,
  rules: [
    {
      ruleId: string,
      source: "learned" | "standing_instruction",
      communicationDomain: string,
      ruleType: string,
      ruleText: string
    }
  ]
}
```

Keep the existing `getMessageLearningContext()` function for backward compatibility by having it call the new function and return only `contextText`.

Rules:

* Preserve current behavior when no rules exist.
* Include only active rules.
* Include standing instructions first.
* Include domain-specific before general.
* Do not expose private data.

## 2. Record Rule Applications

Create a reusable helper:

```ts
recordAgentMailRuleApplications({
  orgId,
  actionId,
  communicationDomain,
  rules
})
```

Insert rows into:

```txt
agentmail_rule_applications
```

Fields:

* org_id
* action_id
* rule_source
* rule_id
* communication_domain
* applied_at

Rules:

* Idempotent per `(action_id, rule_source, rule_id)` if possible.
* Do not crash draft generation if insert fails.
* Log warning and continue.
* Skip insert if no rules.
* Skip insert if no actionId.
* Ensure org scoping.

# Data Integrity

## Idempotency

Avoid duplicate rows if the same draft generation retries.

Preferred:

* Unique index on:

  * `action_id`
  * `rule_source`
  * `rule_id`

If index does not exist, create it safely.

## Failure Safety

Rule tracking must never block:

* draft generation
* approval
* sending
* regeneration
* workflows
* heartbeat

All instrumentation should fail open.
