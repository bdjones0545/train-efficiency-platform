# TrainEfficiency Replit Task

## Audit Athlete Intake Pipeline Workflow

You are working inside the TrainEfficiency codebase.

Your task is to audit the **Athlete Intake Pipeline** workflow before we make changes.

Do **not** redesign the UI yet.

Do **not** add new features yet.

First, trace and document what currently happens when a new athlete or lead enters the pipeline.

---

## Context

In the frontend, each Athlete Intake Pipeline card currently shows something like:

Sophia Stewart
Caydence Partridge
New
Volleyball
Bluffton High School
1d ago
Organic · organic

Buttons:

* Schedule Eval
* Convert to Athlete
* Email
* Archive

We need to understand exactly what these buttons do, what backend routes they call, what database records are created or updated, and what automations are triggered.

---

## Primary Questions

Answer these with evidence from the codebase.

### 1. New Lead Creation

When a new athlete enters the intake pipeline:

* Which frontend component renders the card?
* Which API route fetches the pipeline data?
* Which database table stores the lead?
* What fields are stored?
* How is status determined?
* How are source/channel values determined?
* How is "1d ago" calculated?

---

### 2. Automatic Intro Email

Determine whether a new lead automatically triggers:

* An intro email
* A welcome email
* A parent email
* A coach notification
* An internal task
* A follow-up sequence
* An AI-generated email draft

For each item, document:

* Whether it exists
* Where the code lives
* Whether it is wired to real lead creation
* Whether it is currently mocked, stubbed, or incomplete

---

### 3. Organization Landing Page / Signup Link

Audit whether the system generates a link for the athlete or parent to sign up.

Check:

* Is there an org landing page route?
* Is there an athlete signup route?
* Is there a public intake form route?
* Does the email include a link?
* Is the link org-specific?
* Does it prefill lead data?
* Does it create an athlete account?
* Does it convert a lead to an athlete automatically after signup?

Document all existing routes and gaps.

#### Convert to Athlete

* Frontend click handler

* Whether a template is used

#### Archive

Find:

* Frontend click handler
* API route called
* Whether this is a soft archive or delete
* Database fields updated
* Whether archived leads can be restored
* Whether archive reason is captured

## Important Rules

* Do not make assumptions.
* Cite exact file paths and function names.
* Distinguish clearly between:

  * implemented and wired
  * implemented but not wired
  * mocked/stubbed
  * missing
* Do not modify files during this audit unless explicitly asked.
* If you find broken buttons, dead routes, missing handlers, or placeholder data, call them out clearly.
* Focus on workflow truth, not ideal product behavior.

The goal is to understand what actually happens today before we harden the pipeline later with Claude Code.
