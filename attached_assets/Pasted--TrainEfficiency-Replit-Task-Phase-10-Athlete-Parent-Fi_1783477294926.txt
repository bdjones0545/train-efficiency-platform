# Goal

When an athlete or parent logs in for the first time, they should not land in an empty app.

They should see a simple onboarding experience that explains:

* Who they are connected to
* What has already been completed
* What still needs action
* What happens next
* How to contact the coach or organization

This is **not** a full athlete portal redesign.

This is a first-login onboarding layer.

---

# Core Experience

Create a lightweight first-login dashboard for athletes and guardians.

Suggested routes:

```text
/client/onboarding
/guardian/onboarding
```

or reuse existing authenticated client routes if they already exist.

The page should answer:

> “What do I need to do before training starts?”

---

# User Types

## Athlete

Show:

* Profile summary
* Organization name
* Sport / position / school / grade
* Next session if scheduled
* Program assignment status
* Readiness checklist
* Contact coach / organization action
* Account setup completion

## Guardian / Parent

Show:

* Linked athlete or athletes
* Each athlete's onboarding readiness
* Upcoming session if available
* Program assignment status
* Missing items
* Communication preferences
* Contact organization action

---

# Backend Scope

## 1. Authenticated Onboarding Summary

Add route(s):

```text
GET /api/client/onboarding
GET /api/guardian/onboarding
```

Use the authenticated user.

Return only records the user is allowed to see.

For athlete:

```ts
{
  userType: "athlete",
  organization,
  athlete,
  profile,
  onboardingChecklist,
  readiness,
  nextSession,
  assignedProgram,
  missingItems,
  nextBestAction,
  contactOptions
}
```

For guardian:

```ts
{
  userType: "guardian",
  guardian,
  organization,
  linkedAthletes: [
    {
      athlete,
      onboardingChecklist,
      readiness,
      nextSession,
      assignedProgram,
      missingItems,
      nextBestAction
    }
  ],
  communicationPreferences,
  contactOptions
}
```

Rules:

* Athletes can only see their own record.
* Guardians can only see linked athletes.
* No admin-only notes should leak.
* Do not expose internal AI summaries unless explicitly safe.
* Do not expose revenue/billing internals beyond user-safe readiness wording.

---

## 2. First Login Detection

Audit existing user schema.

Look for:

* lastLoginAt
* onboardingCompletedAt
* firstLoginCompleted
* profileCompletedAt
* client onboarding state

If fields exist, use them.

If not, add lightweight fields safely, such as:

```text
users.firstLoginCompletedAt
users.profileConfirmedAt
```

or use a profile/onboarding table if one already exists.

The system should know whether this is the first login.

---

## 3. User-Safe Readiness Mapping

Do not show internal labels like:

* PAIL
* billing blocked
* waiver blocker
* operational readiness score
* onboarding alert severity

Translate into friendly language:

Internal:

```text
needs_program
```

User-facing:

```text
Your coach is preparing your training plan.
```

Internal:

```text
needs_first_session
```

User-facing:

```text
Your first session still needs to be scheduled.
```

Internal:

```text
needs_billing
```

User-facing:

```text
Payment setup is needed before training starts.
```

Internal:

```text
ready_to_train
```

User-facing:

```text
You're ready for your first training session.
```

## 5. Contact Organization

Provide a simple contact action.

Options:

* Create AgentMail draft / message to organization
* mailto fallback only if no AgentMail user-facing pathway exists
* existing support/contact route if available

Preferred:

```text
Contact Coach
```

This should create or open an AgentMail-backed conversation if supported.

If not supported, link to org contact email.
