## Token storage rules

Minimum:

```text

```


* in the Slack interactive button value

Use a server-side pepper or HMAC key if the repository already has an appropriate secret-management pattern.

If using plain SHA-256, ensure token entropy is sufficient to prevent offline guessing.

# Part 5 — Atomic token consumption

Token consumption must be atomic and single-use.

Use a database transaction or conditional update equivalent to:

```sql
UPDATE slack_action_tokens
SET
  status = 'processing',
  processing_at = NOW()
WHERE
  token_hash = ?
  AND status = 'pending'
  AND expires_at > NOW()
  AND org_id = ?
  AND slack_team_id = ?
  AND slack_user_id = ?
RETURNING *;
```

The exact SQL should follow repository conventions.

Only one request may transition:

```text
pending → processing
```

If no row is returned, classify the reason safely:

```text
invalid
expired
already_consumed
already_processing
canceled
wrong_user
wrong_team
wrong_org
```

Do not reveal unnecessary internal details to the Slack user.

## Execution lifecycle

After claiming the token:

```text
processing
→ execute canonical TrainEfficiency action
→ consumed
```

If the canonical business action fails:

```text
processing
→ failed
```

Store a sanitized error.

Do not automatically reset a failed token to pending unless the action is proven safe to retry.

For ambiguous execution outcomes, fail closed and require a new confirmation preview.

## Duplicate Slack retries and double-clicks

Test and enforce:

* two simultaneous clicks execute the business action once
* Slack retry delivery does not execute twice
* a consumed token cannot be reused
* a processing token cannot be claimed by another request
* an expired token cannot execute
* a token issued to User A cannot be used by User B
* a token issued for Org A cannot be used for Org B
* a token issued in Team A cannot be used in Team B
