# Phase 2 Goal

Make **Convert to Athlete** actually convert a lead into an athlete.

Current behavior:

* Clicking Convert to Athlete only sets:

  * `bookingStatus = "enrolled"`
  * `convertedAt = now`

Target behavior:

* Clicking Convert to Athlete should start a real onboarding workflow:

  * Create or link an athlete/customer record.
  * Preserve the original lead submission.
  * Transfer relevant intake data.
  * Create onboarding/invitation communication.
  * Initialize athlete intelligence readiness where appropriate.
  * Keep the lead pipeline and athlete records synchronized.

---

# Architectural Rules

## AgentMail

Use **AgentMail** for human/coach/customer communication:

* Welcome from coach
* Parent/guardian follow-up
* Evaluation follow-up
* Manual onboarding message
* AI-generated relationship-based messages

## Infrastructure Email

Infrastructure/system emails may remain outside AgentMail:

* Account invitation
* Magic link
* Password reset
* Email verification
* Security alert
* Payment receipt

Do not migrate those in this phase unless the existing app already routes them through a system email service.

# Scope

## 1. Replace Hollow Convert Action

Current behavior:

```ts
PATCH /api/admin/athlete-leads/:id
{
  bookingStatus: "enrolled",
  convertedAt: now
}
```

Target behavior:

Convert should call a dedicated onboarding route or service.

Suggested endpoint:

```ts
POST /api/admin/athlete-leads/:id/convert
```

This route should orchestrate the conversion workflow.

Avoid overloading the generic PATCH route with complex business logic.

---

## 2. Create or Link Athlete Record

Inspect the existing schema before implementing.

Find the correct existing athlete/customer/user table.

Possible names may include:

* `athletes`
* `users`
* `clients`
* `organizationMembers`
* `athleteProfiles`
* `teamTrainingAthletes`
* `customers`

Do not create a duplicate athlete table if one already exists.

Conversion should:

* Create a real athlete/client/customer record if one does not already exist.
* Link the new athlete record back to the original `leadCaptureSubmissions` row.
* If an athlete with the same org + email already exists:

  * Do not duplicate.
  * Link the lead to the existing athlete.
  * Return a clear result such as `created: false`, `linkedExisting: true`.

Transfer relevant fields:

* Athlete name
* Email
* Phone
* Age
* Grade
* Sport
* Position
* School
* Goals
* Experience level
* Current training status
* Commitment level
* Parent/guardian name
* Source / UTM metadata where appropriate
* Notes

## 3. Lead Status + Intelligence Sync

After successful conversion:

Update `leadCaptureSubmissions`:

* `bookingStatus = "enrolled"`
* `convertedAt = now`
* Link field to athlete/user if one exists, such as:

  * `linkedUserId`
  * `athleteId`
  * or the appropriate existing schema field

Update `leadIntelligenceProfiles`:

* `pipelineStage = "converted"`
* Add stage transition audit entry using existing stage transition conventions.
* Preserve existing processing logs and history.

Do not lose prior AI summary, draft, or follow-up history.

## 4. Onboarding Communication

After conversion, create an onboarding communication.

This should be **AgentMail-backed** if it is a human-style welcome or coach message.

Example:

* Subject: "Welcome to [Organization Name]"
* Body:

  * Congratulate the athlete/parent
  * Explain next steps
  * Include organization-specific onboarding or landing page link if available
  * Mention evaluation/training start details if known
  * Encourage reply with questions

Important:

* Do not auto-send unless existing AgentMail workflow requires explicit approval.
* Prefer draft/approval queue first.
* Associate the draft with:

  * lead submission
  * athlete record if possible
  * org
  * conversation/contact identity

If the app already has account invitation infrastructure, that can be created separately as a system email.

---

## 5. Account Invitation / Portal Access

Audit existing auth/account invitation capabilities.

If there is already a supported flow for inviting athletes/users:

* Create or queue an account invitation.
* Link the invited user to the athlete record.
* Mark invitation status appropriately.

If no supported flow exists:

* Do not invent a full auth system.
* Leave a clear TODO / response field.

Expected conversion response should clearly report:

```ts
{
  athleteCreated: true,
  linkedExisting: false,
  accountInviteCreated: true | false,
  welcomeDraftCreated: true | false,
  pailInitialized: true | false
}
```

## 6. Initialize Athlete Intelligence / PAIL
