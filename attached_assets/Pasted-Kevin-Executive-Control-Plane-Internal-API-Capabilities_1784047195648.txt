Kevin Executive Control Plane — Internal API, Capabilities, Permissions, and Agent Execution
Act as a world-renowned full-stack software architect, AI systems engineer, security engineer, and production SaaS operator.
TrainEfficiency already contains Kevin integration infrastructure, platform agents, AgentMail, the CEO Agent, approval workflows, background workers, organization-level permissions, audit records, and operational services.
Kevin is hosted externally inside an always-on virtual computer. He can reason, retain executive memory, analyze the platform, and communicate through Slack and TrainEfficiency. However, he currently cannot reliably perform meaningful actions inside TrainEfficiency.
The goal of this project is to build the secure Executive Control Plane that allows Kevin to operate TrainEfficiency through explicit, authenticated, organization-scoped capabilities.
This is not a request to give Kevin broad access to the Replit environment, database, or internal code.
Kevin must only interact with TrainEfficiency through a controlled internal API that validates every requested action, determines the permitted execution mode, calls existing platform services, verifies the result, and records a complete audit trail.
Core Principle
The internal API is the permission boundary.
Kevin VM
    ↓ signed and authenticated intent
TrainEfficiency Executive Control Plane
    ↓
Capability Registry
    ↓
Policy and Permission Engine
    ↓
Approval or Execution
    ↓
Existing TrainEfficiency Service
    ↓
Verification
    ↓
Outcome and Audit Record
    ↓
Result returned to Kevin
Kevin must never receive unrestricted database access.
Kevin must never directly modify another agent’s memory.
Kevin must never bypass existing business services by writing directly to tables.
Kevin must never be given one global “can manage TrainEfficiency” permission.
Every action must be represented as an explicit capability with its own risk, permission, approval, execution, and verification policy.

Execution-mode behavior:
disabled
→ reject with a typed policy denial

observe
→ permit read-only context retrieval only

recommend
→ produce a recommendation or execution plan without creating a platform-side action

draft
→ create a reversible draft or proposed task without external side effects

→ persist the proposed action and create a scoped approval request

→ execute only within configured limits and only after every policy check passes
