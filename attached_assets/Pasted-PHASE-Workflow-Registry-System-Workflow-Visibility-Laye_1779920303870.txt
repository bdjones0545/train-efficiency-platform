PHASE: Workflow Registry + System Workflow Visibility Layer

We are evolving the orchestration builder from a blank-canvas workflow editor into a true operational AI workflow platform. The builder must now understand:

1. Built-in system workflows
2. Organization-created workflows
3. Workflow templates
4. Duplicate/conflict detection
5. Workflow execution analytics
6. Protected infrastructure workflows

This phase should NOT break the current orchestration builder. Extend the architecture cleanly.

====================================================

- clones workflow_definition
- sets:
  source = org_custom
  protected = false
  editable = true
  cloned_from_workflow_id = original
- appends "(Custom)" to title
- opens in builder editor

====================================================
PART 5 — DUPLICATE / CONFLICT DETECTION
====================================================

Build a workflow conflict engine.

Before saving or enabling a workflow:

Analyze:
- trigger overlap
- action overlap
- same pipeline targets
- same email actions
- same scheduling actions
- recursive trigger loops

Detect conflicts like:

Meta Lead
→ AI Outreach
→ Send Email

already exists in another enabled workflow.

Show warning modal:

"An active workflow already handles:
Meta Lead → AI Qualification → Gmail Outreach

Choose:
- View Existing Workflow
- Clone Existing Workflow
- Disable Existing Workflow
- Continue Anyway"

Store conflict results in:
workflow_conflicts table
