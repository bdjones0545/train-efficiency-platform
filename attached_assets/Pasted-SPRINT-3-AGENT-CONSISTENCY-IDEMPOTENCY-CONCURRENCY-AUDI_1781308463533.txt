SPRINT 3 — AGENT CONSISTENCY, IDEMPOTENCY & CONCURRENCY AUDIT

Objective:

Sprint 1 established org isolation.

Sprint 2 established agent execution integrity.

Sprint 3 establishes execution consistency under concurrency, retries, crashes, webhook replays, network failures, and duplicate-trigger scenarios.

Goal:

No agent, workflow, email, webhook, heartbeat, approval, billing action, or retry path should be capable of creating duplicate side effects.

──────────────────────────────
PHASE 1 — WORKFLOW JOB IDEMPOTENCY
──────────────────────────────

Audit:

workflow_jobs
agent_execution_locks
workflow orchestrator
job retry paths

Verify:

1. Can the same workflow job execute twice?
2. Can a retry create duplicate side effects?
3. Can a crashed worker re-run completed work?
4. Can concurrent workers claim the same job?

For every workflow type:

* identify idempotency key
* identify lock strategy
* identify retry strategy

Return:

* duplicate execution risks
* missing idempotency keys
* lock weaknesses

──────────────────────────────
PHASE 2 — HEARTBEAT CONSISTENCY
──────────────────────────────

CEO Heartbeat

Verify:

2. Can startup + manual run collide?
3. Can startup + cron collide?
4. Can heartbeat actions execute twice?
5. Can priorities be duplicated?
6. Can recommendations be duplicated?
7. Can approvals be generated twice?

Verify lock behavior:

${orgId}:${jobName}:${timeBucket}

Confirm it prevents overlap.

Return:

* overlap risks
* duplicate generation risks
* lock validation
